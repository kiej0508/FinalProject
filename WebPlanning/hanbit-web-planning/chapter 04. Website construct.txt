1. UML 작성
	1-1. UML(Unified Modeling Language)이란?
		- 프로젝트의 전체적인 이해를 도와 같은 목표를 향하도록 하는 유용한 도구
		- 프로젝트 초기에 고객의 이동 동선과 필요한 기능의 흐름을 세세하게 정의
		
		1) 객체지향적 설계는?
			- 객체 : 정체성 = 유일성??
			- 유스케이스, 클래스, 객체, 시퀀스, 스테이트차트 등의 사례가 있음
			
		2) UML 도구
		
		3) UML 구성 요소 이해하기
			- 아 이거 생긴거 알아둬야 나중에 남이 쓴거 이해하고 내가 쓰고 하겠구나
			(1) 구조 요소
				A. 클래스 : 비슷한 속성과 공통적인 행동을 가지는 사물의 범주 혹은 그룹으로 객체를 생성하는 템플릿 역할
				B. 유스케이스 : 시스템이 사용하는 시나리오의 모음으로 유스케이스는 시스템이 사용자에게 어떻게 비추어질지 나타냄
				C. 행위자 : 유스케이스를 시작하게 하거나 유스케이스의 결과를 얻는 개체(시스템 혹은 사람)
			(2) 관계
				A. 연관 : 두 클래스 사이의 관계
				B. 집합연관 : 한 클래스가 다른 전체의 부분인 형태를 가지는 연관으로서, 집합체 클래스는 한 개 이상의 컴포넌트 클래스로 구성
				C. 복합연관 : 강한 집합연관에 의해 만들어진 클래스로 각각의 컴포넌트 클래스가 오직 하나의 전체 클래스만 속할 수 있음
			(3) 확장
				A. 스테레오 타입 : 기존의 UML 요소를 취하려 해당 상황에 맞는 요소를 새로 만들 수 있게 하는 구조로 <>를 사용
				B. 제약 : UML 요소가 가지는 의미를 확장하는 수단으로서, UML 다이어그램에 직접 추가하는 규칙으로 {}로 나타냄
				
		4) 기획자가 진행해야 할 다이어그램
			- 기획자 : 유스케이스, 액티비티, 다이어그램 제작
			- 개발자 : 클래스, 시퀀스, 스테이트차트 다이어그램 제작
			(1) 유스케이스 다이어그램
				 - 사용자의 요구사항을 차악하고 키워드로 표현하는 것
				 - 사용하는 사람 또는 사물의 행위로 바꿔 정의할 수 있음
				 A. 정의
					- 사용자 관점에서 SW 시스템의 범위와 기능 정의
					- 시스템이 해야 할 '무엇'을 작성
					- '어떻게'는 서술하지 않음
				B. 목적
					- 업무 범위 정의
					- 사용자 정의
					- 업무 기능 정의
					- 사용자 요구사항 정의
					- 사용자와 개발자 간 의사소통 도구
					- 분석, 설계 작업 기준
					- 테스트 기준
				C. 구성 요소
					- Association : 액터와 유스케이스 간 관계
					- Generalization : 액터끼리, 유스케이스끼리의 관계로 일반화 관계 정의
					- Include : 한 유스케이스가 다른 유스케이스에게 서비스를 요청하는 관계. 서비스는 반드시 수행
					- Extend : 한 유스케이스가 다른 유스케이스에게 서비스를 요청하는 관계. 서비스는 조건에 따라 수행
			(2) 액티비티 다이어그램
				- 시스템의 처리 흐름을 규정
				- 작업자가 작업의 흐름을 손쉽게 알아볼 수 있도록 가이드
				- 유스케이스 다음에 정의
				A. 정의
					- 처리 로직이나 조건에 따른 처리 흐름을 순서에 따라 정의한 모델(=플로차트)
				B. 목적
					- 처리 순서 표헌(업무 프로세스 정의 시점)
					- 비즈니스 프로세스 정의 : As is, To be 분석
					- 프로그램 로직 정의
					- 유스케이스 실현
				C. 작성 순서
					- 작성 대상 선정 : 업무 프로세스 모델링, 오퍼레이션 사양 정의
					- Swim lane : 대상 영역에 명확한 역할을 정의해야할 때
					- 처리 절차 모델링 : 시작점, 끝점 반드시 표현
				D. 구성 요소
					- Activity : 행위나 작업
					- Transition : 액티비티가 행위를 완료하고 다른 액티비티로 처리 순서 이동
					- Synchronization bar : 병렬처리 절차가 시작되거나 모이는 지점
		5) UML 요약
			- UML = 수행하고 있는 프로젝트를 작은 객체 단위로 단위하하고 객체를 하나씩 완성해가며 수행의 성과를 높일 수 있는 방법
